c++11-14中的有用特性1

被弃用的主要特性

  • 弃用不等于废弃，只是用于暗示程序员这些特性将从未来的标准中消失，应该尽量避免使用。
  • 但是，已弃用的特性依然是标准库的一部分，并且出于兼容性的考虑，这些特性其实会『永久』保留。

  1. 如果一个类有析构函数，为其生成拷贝构造函数和拷贝赋值运算符的特性被弃用了。

  2. 不再允许字符串字面值常量赋值给一个 char *。

    • 如果需要用字符串字面值常量赋值和初始化一个 char *，应该使用 const char * 或者 auto。

  3. bool 类型的 ++ 操作被弃用

  4. C 语言风格的类型转换被弃用，应该使用 static_cast、reinterpret_cast、const_cast 来进行类型转换.

  5. auto_ptr 被弃用，应使用 unique_ptr。

  6. register 关键字被弃用。

  7. C++98 异常说明、 unexpected_handler、set_unexpected() 等相关特性被弃用，应该使用 noexcept。

与C的兼容性

  • 在编写 C++ 时，也应该尽可能的避免使用诸如 void* 之类的程序风格
  • 而在不得不使用 C 时，应该注意使用 extern "C" 这种特性，将 C 语言的代码与 C++ 代码进行分离编译，再统一链接这种做法。

nullptr 与 constexpr

  • nullptr：专门用来区分空指针、0，目的是为了替代 NULL。

    • nullptr 的类型为 nullptr_t，能够隐式的转换为任何指针或成员指针的类型，也能和他们进行相等或者不等的比较。
    • 传统 C++ 会把 NULL、0 视为同一种东西，这取决于编译器如何定义 NULL，这将导致了 C++ 中重载特性会发生混乱。
  • constexpr：让用户显式的声明函数或对象构造函数在编译器中会成为常数，用于定义数组等工作；
    • 这个关键字明确的告诉编译器应该去验证增加了该关键字的函数，在编译时就应该是一个常数。

类型推导：auto 和 decltype

  • auto 的使用：
  • 要注意的点：auto 不能用于函数传参。
  • 要注意的点：auto 还不能用于推导数组类型。

  • decltype 关键字是为了解决 auto 关键字只能对变量进行类型推导的缺陷而出现的。

    • decltype 可以对表达式的类型进行推导，却不实际计算表达式的值；
    • 用法和 sizeof 很相似：decltype(表达式)；auto x = 1; auto y = 2;decltype(x+y) z; z是int类型。

  • decltype 关键字可用于尾返回类型（trailing return type）

    • 写法比较别扭，还好C++14开始直接使用 auto关键字推导函数的返回值类型。

模板增强

  • 传统 C++ 中，模板只有在使用时才会被编译器实例化。
  • 换句话说，只要在每个编译单元（文件）中编译的代码中遇到了被完整定义（给与了实际类型的）的模板，都会实例化。
  • 这就产生了重复实例化而导致的编译时间的增加，并且，我们没有办法通知编译器不要触发模板实例化。
  • typedef 关键字无法用于模板，而是用using关键字就可以

面向对象增强

  • 引入了委托构造：使得可以在同一个类中由一个构造函数调用另一个构造函数。
  • 使用using 关键字引入继承构造函数概念：传统写法中派生类的构造函数如果需要继承是需要将参数一一传递的，也就是重新写一遍派生类自己的构造函数。
  • 引入了 override ，进行显示虚函数重载，避免派生类具有与基类虚函数同名的成员函数，而该派生类成员函数被隐式编译为虚函数。
  • 引入了final 则是为了防止类被继续继承以及终止虚函数继续重载引入的。
